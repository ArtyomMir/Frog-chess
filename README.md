# Frog Pond Duel

Собственный учебный проект по заданию **Frog Chess** из `greenswamp/indiv/frogchess`. Это веб-сервис для партии 1 на 1: сервер создает матч, подключает второго игрока, генерирует стартовую доску, проверяет правила прыжков и рассылает обновления состояния.

## Что реализовано

- матч для двух игроков;
- случайная серверная расстановка 36 лягушек на внутренних клетках доски 8x8;
- по 18 лягушек каждого цвета;
- обязательное удаление одной любой лягушки перед первым прыжком игрока;
- одиночные и цепочные прыжки по горизонтали, вертикали и диагонали;
- проверка занятости клетки приземления и наличия перепрыгиваемой лягушки;
- удаление перепрыгнутых лягушек;
- механика болота по внешнему краю доски;
- автоматический пропуск хода, если у игрока нет допустимых прыжков;
- завершение игры после двух пропусков подряд;
- победа игрока, совершившего последний прыжок;
- обработка отключения игрока через SignalR с окном переподключения 45 секунд;
- браузерный интерфейс для локальной проверки.

## Архитектурное Решение

Выбран гибридный подход: игровые команды идут через REST, а обновления состояния рассылаются через SignalR. REST удобен для явной HTTP-семантики, простого тестирования и понятных ошибок `400/403/404/409`. SignalR нужен для уведомлений о ходе соперника, подключении, отключении и завершении матча без ручного обновления страницы.

Чистый SignalR не выбран, потому что при таком подходе сложнее проверять команды обычным HTTP-клиентом и писать небольшие интеграционные сценарии. Чистый REST polling не выбран как основной механизм, потому что для пошаговой игры удобнее получать изменения сразу. При этом фронтенд оставляет polling как запасной вариант, если клиентская SignalR-библиотека не загрузилась.

## Структура

```text
FrogPondDuel.Core/     чистая игровая модель и проверка правил
FrogPondDuel.Web/      ASP.NET Core REST API, SignalR Hub, статический frontend
FrogPondDuel.Tests/    консольный test runner без внешних NuGet-пакетов
```

## Запуск

Требуется .NET SDK 10.

```powershell
dotnet build .\FrogPondDuel.slnx
dotnet run --project .\FrogPondDuel.Web\FrogPondDuel.Web.csproj --urls http://127.0.0.1:5094
```

После запуска откройте:

```text
http://127.0.0.1:5094
```

Для игры вдвоем откройте страницу в двух браузерах или в обычной и приватной вкладке. Первый игрок создает матч и передает второму `ID матча`.

## Тесты

```powershell
dotnet build .\FrogPondDuel.slnx
dotnet .\FrogPondDuel.Tests\bin\Debug\net10.0\FrogPondDuel.Tests.dll
```

Покрыты:

- генерация стартовой расстановки;
- правило первого удаления;
- прыжки через свои и чужие лягушки;
- цепочный прыжок;
- недопустимые прыжки;
- запрет хода не в свою очередь;
- механика болота;
- автопас и конец игры;
- конкурентная отправка двух ходов.

## API

```http
GET  /api/health
POST /api/games
POST /api/games/{matchId}/join
GET  /api/games/{matchId}?playerToken={token}
POST /api/games/{matchId}/remove
POST /api/games/{matchId}/jumps
POST /api/games/{matchId}/pass
```

Realtime endpoint:

```text
/gameHub
```

Клиент вызывает `WatchGame(matchId, playerToken)` и получает событие `stateChanged`.

## Примеры Команд

Создать игру:

```json
{ "playerName": "Alice" }
```

Войти в игру:

```json
{ "playerName": "Bob" }
```

Удалить первую лягушку:

```json
{
  "playerToken": "00000000-0000-0000-0000-000000000000",
  "row": 3,
  "col": 4
}
```

Отправить цепочку прыжков:

```json
{
  "playerToken": "00000000-0000-0000-0000-000000000000",
  "path": [
    { "row": 3, "col": 1 },
    { "row": 3, "col": 3 },
    { "row": 3, "col": 5 }
  ]
}
```

## Примечания

Состояние хранится только в памяти сервера. После перезапуска все матчи исчезают, что допустимо по условиям задания. Игрок идентифицируется случайным `Guid`-токеном, который выдается при создании или входе в матч.
